## Verdict général

**Note : 5,5 / 10.**

Trois phrases brutales :

1. **La config fonctionne mais elle est devenue une décharge de sédiments** — chaque session a déposé sa couche sans enlever l'ancienne, et aujourd'hui les mêmes règles responsive sont dupliquées **à 7 endroits** avec des formulations légèrement différentes (dérive assurée à moyen terme).
2. **L'agent ne peut PAS travailler en autonomie parce que le design ne le permet pas** : le CLAUDE.md racine commande `bash .claude/scripts/check-open-prs.sh` comme action OBLIGATOIRE au début de chaque session, or **ce script n'existe pas** — chaque session démarre donc sur un warning silencieux. Trois autres chemins de fichiers référencés n'existent pas non plus.
3. **Le workflow "1 PR = 1 bloc cohérent" est une bonne idée documentée 4 fois**, mais tant qu'il n'y a pas de queue de tâches structurée (`.claude/work/` est un bazar de 5 fichiers qui se marchent dessus), l'agent ne peut pas l'appliquer en chaînant tout seul — il revient toujours à Romeo pour savoir quoi faire ensuite. C'est le vrai goulot d'étranglement.

Ce que Stripe/Linear/Shopify/Vercel font différemment — et qu'on n'a pas ici — est détaillé en fin de rapport.

## Ce qui marche

Liste courte, zéro flatterie, juste les faits :

- **Agents spécialisés clairement séparés** : dev, reviewer, verify, ops, writer, market. Chacun a une mission précise, des outils limités, une section "TU NE FAIS PAS". Standard de l'industrie.
- **Hooks `settings.json` effectivement bloquants** : 14 PreToolUse qui bloquent les bêtises réelles (`--no-verify`, push sur main, `any` TypeScript, PR vers main, `pnpm dev`). Bien conçus.
- **Pre-commit husky simplifié à <1s** (prettier + docs auto + lint-staged). Pragmatique, pas de sur-validation locale puisque CI valide.
- **Commit-msg validation** : regex stricte pour `[APP-DOMAIN-NNN]`. Bien pour la traçabilité.
- **Workflow scratchpad activement utilisé** (25+ fichiers en 3 jours). Ce n'est pas du théâtre.
- **CI `quality.yml`** : cache Next.js, filtrage par app changée, commentaires PR automatiques. Du bon niveau.
- **`PROTECTED_FILES.json`** : concept solide, bien structuré (protected / requiresReview / criticalFiles / autoGenerated).
- **Règles métier précises et motivées** : chaque règle stock-triggers-protected cite un bug historique (28 novembre 2025) — c'est exactement comme ça qu'on fait en entreprise.

**C'est le squelette d'une bonne config.** Le problème, c'est tout le reste qui s'est ajouté dessus.

### D. Trous critiques (ce qui manque pour l'autonomie)

C'est ICI que se joue la question "pourquoi l'agent ne peut pas tourner sans Romeo".

#### D.1. Pas de queue de tâches consommable automatiquement

État actuel de `.claude/work/` :
```
ACTIVE.md                 369+ lignes   — backlog + faits, tout mélangé
AGENT-ENTRY-POINT.md      196 lignes    — mode d'emploi agent (créé hier)
NEXT-SPRINTS.md           ~200 lignes   — plan tactique (créé hier)
PROMPTS-TO-COPY.md        ~100 lignes   — prompts pour Romeo (créé hier)
plan-canaux-de-vente.md   ?             — vieux plan
```

Cinq fichiers qui parlent tous du "quoi faire maintenant". Aucun format standardisé. Aucune métadata machine-lisible (YAML frontmatter). L'agent doit lire 1000 lignes de prose pour comprendre quelle est la prochaine tâche.

**Chez Linear / GitHub Projects** : une issue = un fichier YAML/JSON avec `status: todo`, `priority: P1`, `blockers: []`, `playbook: migrate-page-responsive`, `estimated: 2h`. L'agent `pop` la suivante, l'exécute, la passe à `done`. Zéro prose.

#### D.2. Pas de "playbooks" réutilisables

Chaque sprint `BO-UI-RESP-0XX` dans `NEXT-SPRINTS.md` redécrit le même workflow (`git checkout`, commit, push, tests playwright, etc.). Copier-coller partout.

**Ce qui manque** : `playbooks/migrate-page-responsive.md` = recette unique référencée par chaque tâche via `playbook: migrate-page-responsive`. Tu changes la recette une fois, toutes les tâches l'adoptent.

#### D.3. Pas de définition claire "l'agent peut / l'agent doit attendre"

Le CLAUDE.md a une section AUTORISATIONS (créée 2026-04-18) qui s'y essaie, mais elle est mêlée aux règles générales. Et elle est contradictoire avec IDENTITE ("tu ne codes JAMAIS").

**Ce qu'il faut** : un fichier unique `.claude/config/autonomy-boundaries.md` qui liste sans ambiguïté :
- Feu vert (l'agent fait, ne demande pas)
- Feu orange (l'agent propose, demande confirmation laconique)
- Feu rouge (l'agent refuse jusqu'à ordre explicite)

#### D.4. Pas de hook de "session-start" qui charge la prochaine tâche

Le hook `session-context.sh` existe et fait un résumé — mais il ne **nomme pas la prochaine tâche à faire**. L'agent démarre sans direction.

**Ce qu'il faudrait** : un `next-task.sh` qui lit la queue `.claude/queue/01-TODO/` (triée par priorité), affiche la 1ère tâche, et propose "tu veux que je la prenne ?". Ça transforme "qu'est-ce que je fais ?" en "je fais X ou tu veux autre chose ?".

#### D.5. Pas d'auto-review sur les PRs

Les PRs ne déclenchent aucune action automatique de `reviewer-agent`. Le workflow `quality.yml` fait du build/type-check/lint, mais pas d'audit qualité structuré. Conséquence : Romeo doit lancer manuellement `/review` avant chaque PR. S'il oublie, rien ne le rappelle.

**Chez GitHub / CodeRabbit / Copilot** : un bot commente la PR dès l'ouverture avec un résumé + flags. Ici, rien.

#### D.6. Pas de webhook/notification pour les événements GitHub

Quand une PR est mergée, Vercel déploie, mais Claude Code ne le sait pas. L'agent ne peut pas enchaîner "PR mergée → passer à la tâche suivante dans la queue" parce qu'il n'a pas le signal.

#### D.7. Pas de workflow de rebase / conflict resolution

`auto-sync-with-main.sh` bloque les commits si la branche est en retard. Mais il ne rebase pas tout seul. L'agent doit manuellement faire `git fetch && git merge origin/staging`. Avec 5 branches actives en même temps (ce que Romeo fait), c'est Romeo qui finit par gérer.

#### D.8. Pas de documentation "comment choisir quel agent invoquer"

CLAUDE.md liste les 6 agents. Mais ne donne pas d'arbre de décision "si ci, alors ça". Le coordinateur doit deviner. Ça marche pour les cas évidents, pas pour les cas ambigus.

**Exemple ambigu** : "Le logo ne s'affiche pas sur la landing" → `dev-agent` pour le code ? `oneshot` skill ? `reviewer` pour comprendre pourquoi ? `writer-agent` si c'est un problème de doc sur le logo ? Pas de règle claire.

#### D.9. Pas de monitoring de dérive

Personne ne surveille si `.claude/INDEX.md` ment (c'est le cas — voir A.7/A.8/B.8). Pas de script qui vérifie que les fichiers listés dans INDEX.md existent vraiment.

**Facile à corriger** : un test `tests/claude-config.spec.ts` qui fait :
```bash
grep -oE '\.claude/[^ `)]+' CLAUDE.md .claude/**/*.md | while read path; do
  test -e "$path" || echo "CASSE: $path"
done
```

#### D.10. Pas de lien clair avec Linear / GitHub Issues

Les Task IDs `[BO-UI-RESP-001]` sont une convention locale. Rien ne les lie à des issues GitHub ou un outil de ticketing. Conséquence : impossible de prioriser avec un outil externe, impossible de notifier quand une tâche change.

**Principes du design** :

1. **Un fichier = une responsabilité**. Pas de mélange backlog/faits/sprints dans le même fichier.
2. **Machine-lisible > prose**. Tâches en YAML frontmatter pour que l'agent parse sans ambiguïté.
3. **Source unique**. Chaque règle existe à un seul endroit, tout le reste la référence.
4. **Queue FIFO consommable**. L'agent sait exactement quelle tâche prendre en lançant `next-task.sh`.
5. **Playbooks réutilisables**. Une tâche = métadata + `playbook: <nom>`. L'agent lit la recette, pas 200 lignes par tâche.
6. **Décisions versionnées**. `DECISIONS.md` = journal (comme un ADR).

C'est exactement le pattern que **Linear, Stripe, Shopify et Vercel** utilisent pour leurs configs internes d'agents/bots :
- **Linear** : chaque issue est un YAML + body Markdown. Les agents consomment la queue via API. Les workflows sont des playbooks versionnés.
- **Stripe** : les "runbooks" sont dans un dossier séparé des "rules". Les rules changent rarement, les runbooks s'adaptent aux incidents.
- **Shopify** : un fichier `CODEOWNERS` + `CONTRIBUTING.md` par domaine, pas un méga-CLAUDE.md.
- **Vercel** : les ADR (Architecture Decision Records) sont la source de vérité des changements de structure.
